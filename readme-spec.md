## キャッシュ運用方針（tempディレクトリ）

tempディレクトリには、1分おきに取得した音声ファイルをキャッシュとして保存する。
ffmpeg_pullスクリプトは音声を取得しつつ、現在時刻を参照し、.envに記載された保存期間（例：30分、60分など）が経過したファイルを自動的に削除する。
この仕組みにより、不要なキャッシュが残らず、ストレージの効率的な運用が可能となる。

## ruby-server ディレクトリ内の成果物リスト

このプロジェクトで新たに作成・配置する主なファイル、スクリプト、設定例は以下の通りとする。

- main.rb : サーバ本体（Sinatraアプリケーション）
- Gemfile / Gemfile.lock : 必要なgemの定義とロックファイル
- config.ru : Rackサーバ用起動設定
- ffmpeg_pull.rb : ffmpegによるストリーム取得用スクリプト
- README.md : セットアップ・起動手順等の説明書
- .env : 環境変数設定ファイル（必要に応じて）

## 課題の詳細

本問題は、単なるユーザー環境差による時間遅延ではなく、ライブストリームの断続的な切断と再接続に伴い、サーバーおよびクライアント双方に残存するキャッシュが再利用されることで、コンテンツ内の時間軸が非連続にずれる現象である。

特に、エンコーダ停止後にクライアントがタイムアウトする前に再接続が発生した場合、最新データではなく過去のキャッシュが再生されるため、実時間と再生内容の間に数分から数十分規模のズレが生じる。この現象はユーザー環境の違いに依存して顕在化するものの、根本的な要因は配信経路におけるキャッシュおよびバッファ管理の挙動にある。

### 対策案

1. 遅延やずれの最大値を5分程度に抑えるため、バッファ時間を3600秒（1時間）から300秒（5分）程度に短縮する。
2. 時差配信用のデータ取得をicecast経由ではなく、現地Barixから直接取得する方式へ変更する。

#### 現在の構成（push型）

現地Barix → 送信 → icecast → 録音 →（絶対時間処理）→ icecast → user

#### 提案構成（pull型）

現地Barix → 受信 → 録音（soundnesia）→ icecast → user

## 現状と課題

- 36.3.129.234 に Barix Instreamer があり、163.43.157.215（sakuraサーバ）で受信・配信を行っている。
- 現在は icecast を利用しているが、同じ音声が1時間遅れで繰り返し配信される現象が発生している。
- この問題を解決するため、新たな配信システムを構築する。

## 目的

- Instreamer からの音声をリアルタイムで受信し、Web上でライブ配信を行う。
- 同時に、指定した遅延（初期値30分）を設けた音声もWebで配信可能とする。

## 構成案

1. Instreamer → Ruby（TCP受信、ffmpegで処理）
2. Ruby（Sinatra）→ Web公開（ライブ配信）
3. Ruby（Sinatra）→ Web公開（x分遅延配信、初期値30分）
