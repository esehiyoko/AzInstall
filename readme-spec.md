## キャッシュ運用方針（tempディレクトリ）

tempディレクトリには、1分おきに取得した音声ファイルをキャッシュとして保存する。
ffmpeg_pullスクリプトは音声を取得しつつ、現在時刻を参照し、.envに記載された保存期間（例：30分、60分など）が経過したファイルを自動的に削除する。
この仕組みにより、不要なキャッシュが残らず、ストレージの効率的な運用が可能となる。


## ffmpeg_pull.rb の動作仕様


ffmpeg_pull.rbは、30秒ごとにストリームから音声ファイル（mp3）を取得し、tempディレクトリに保存する。
ストリーム取得に失敗した場合や、出力ファイルが異常に小さい場合は、自動的に無音mp3で補われる。
この補填処理は強制的に上書きされるため、常に30秒ごとに正常なmp3ファイルが揃う。
また、成功・失敗・異常時の詳細はログに出力される。

## Sinatraアプリのストリーミング仕様

/live.mp3 および /delay.mp3 では、tempディレクトリ内の音声断片ファイル（audio_*.mp3）を「順番に、無限にストリーミング」する。
クライアントが接続している間、新しい断片が生成されるたびに自動的に次のファイルが流れ続ける。
（20個だけ連結するのではなく、常に最新の断片まで順次再生される）


## HTTPS（TLS）対応方針

### Webサーバー

アプリケーションサーバーには **Puma** を使用する。
CentOS 7 では thin（EventMachine依存）のネイティブ拡張ビルドが失敗しやすいため、Rubyの標準スレッドで動作し信頼性の高い Puma を採用する。

### TLS の終端

Sinatra（Rack）自体はTLSを終端しない。TLSは以下のいずれかで処理する。

#### 方式A: Puma に直接証明書を渡す（シンプル構成向け）

`main.rb` が `server.crt` / `server.key` の存在を確認し、あれば自動的に HTTPS で起動する。
なければ HTTP（tcp://）にフォールバックする。

```sh
# bootstrap.rb 実行後、server.crt / server.key が生成されていれば自動で HTTPS 起動
ruby main.rb
```

クライアントが独自の証明書を持ち込む場合は、`server.crt` と `server.key` をそのファイルで**上書き置換**するだけでよい。Sinatra 側のコードは変更不要。

#### 方式B: nginx リバースプロキシ（本番推奨）

nginx が TLS 終端を担い、`localhost:4567` へ HTTP でプロキシする。
この場合 Puma は HTTP のまま動作し、証明書は nginx の設定ファイルで管理する。

### FQDN について

FQDNは **証明書の CN/SAN フィールドに含まれる** ものであり、Sinatra・Puma の設定に記載する必要はない。
`set :bind` はバインドするIPアドレスのみを指定する。

### 証明書の差し替え手順（クライアントが証明書を持ち込む場合）

1. 既存の `server.crt` / `server.key`（bootstrap で生成した自己署名）を削除またはバックアップする。
2. クライアントが用意した証明書ファイルを `server.crt`、秘密鍵を `server.key` という名前でプロジェクトルートに配置する。
3. `ruby main.rb` を起動するだけで HTTPS が有効になる。

## 課題の詳細

本問題は、単なるユーザー環境差による時間遅延ではなく、ライブストリームの断続的な切断と再接続に伴い、サーバーおよびクライアント双方に残存するキャッシュが再利用されることで、コンテンツ内の時間軸が非連続にずれる現象である。

特に、エンコーダ停止後にクライアントがタイムアウトする前に再接続が発生した場合、最新データではなく過去のキャッシュが再生されるため、実時間と再生内容の間に数分から数十分規模のズレが生じる。この現象はユーザー環境の違いに依存して顕在化するものの、根本的な要因は配信経路におけるキャッシュおよびバッファ管理の挙動にある。

### 対策案

1. 遅延やずれの最大値を5分程度に抑えるため、バッファ時間を3600秒（1時間）から300秒（5分）程度に短縮する。
2. 時差配信用のデータ取得をicecast経由ではなく、現地Barixから直接取得する方式へ変更する。

#### 現在の構成（push型）

現地Barix → 送信 → icecast → 録音 →（絶対時間処理）→ icecast → user

#### 提案構成（pull型）

現地Barix → 受信 → 録音（soundnesia）→ icecast → user

## 現状と課題

- 36.3.129.234 に Barix Instreamer があり、163.43.157.215（sakuraサーバ）で受信・配信を行っている。
- 現在は icecast を利用しているが、同じ音声が1時間遅れで繰り返し配信される現象が発生している。
- この問題を解決するため、新たな配信システムを構築する。

## 目的

- Instreamer からの音声をリアルタイムで受信し、Web上でライブ配信を行う。
- 同時に、指定した遅延（初期値30分）を設けた音声もWebで配信可能とする。

## 構成案

1. Instreamer → Ruby（TCP受信、ffmpegで処理）
2. Ruby（Sinatra）→ Web公開（ライブ配信）
3. Ruby（Sinatra）→ Web公開（x分遅延配信、初期値30分）
